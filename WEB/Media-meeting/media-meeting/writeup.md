# Media-meeting - upCTF Writeup

## Challenge Summary
- Challenge name: `Media-meeting`
- Goal: reach the hidden meeting/game condition and retrieve a flag in format `upCTF{...}`.
- Main observation: score-based flag gate, deterministic game logic, and an exploitable online solving strategy.

## Initial Recon
After reading server code (`src/server.js`), the important endpoints are:
- `POST /api/register`
- `POST /api/login`
- `POST /api/play`
- `GET /api/me`
- `POST /api/hint`
- `GET /api/adminPlay` (admin-protected)

Flag disclosure happens in `GET /api/me`:
- If `user.score >= 777`, response includes `flag`.

From `src/server.js`:
- `TARGET_SCORE = 777`
- A correct move gives:
- `cross` -> `69` points
- `wait` -> `1` point
- Wrong move resets score to `0`

So we need to keep winning until score reaches at least `777`.

## How the Game Works
The safe move is generated by:
- `getCorrectMove(userId, numberOfPlays)`
- HMAC-SHA256 over `${userId}:${numberOfPlays}:${GAME_SEED}`
- Last hex nibble parity:
- even -> `cross`
- odd -> `wait`

That means the game is deterministic if `GAME_SEED` is known.

## Local Solve Path (Deterministic)
`docker-compose.yaml` defines:
- `GAME_SEED=WHAT_HAPPENS_IN_M3DIA_STAYS_IN_M3DIA`
- `FLAG=upCTF{esta_flag_é_tão_real_como_as_historias_das_reuniões_de_m3dia}`

With known seed, we can fully predict every safe move and never fail.

### Deterministic Solver Logic
1. Register random user.
2. Login and grab `token` cookie.
3. Decode JWT payload to get `user.id`.
4. For each play index `plays`:
- compute safe move from HMAC formula
- send `POST /api/play` with that action
5. Once score >= 777, call `GET /api/me` and read `flag`.

## Why Remote Needs a Different Approach
On remote instances, environment values can differ from local compose defaults.
In our run:
- Local `GAME_SEED` prediction did not match remote behavior.
- Forging `adminToken` with local `JWT_SECRET` also failed remotely.

So we used a seed-agnostic strategy.

## Fast Remote Solve Strategy (Seed-Agnostic)
Key fact:
- Correct `cross` gives `69` points.
- Need `777` points.
- `12 * 69 = 828` (enough).

So if we get **12 correct `cross` in a row**, we win.

For random parity, each attempt has about 50% chance to be `cross`-safe.
Probability of 12 consecutive wins:
- `1 / 2^12 = 1 / 4096` for any specific 12-window (rough intuition)

To solve quickly under time pressure:
- Run many accounts in parallel.
- Each worker repeatedly submits `cross`.
- Track streak score locally.
- On `score >= 777`, call `/api/me` and print flag.

## Root Cause / Security Notes
- Game security depends entirely on hidden server seed and play state.
- Score resets to zero on fail, but does not prevent high-throughput brute forcing.
- No rate limiting / anti-automation on:
- registration
- login
- gameplay endpoint
- Per-user isolation allows massively parallel trials.

## Suggested Fixes
1. Add strict rate limiting per IP and per account for `register`, `login`, and `play`.
2. Add global anti-abuse controls (burst limits, temporary bans, proof-of-work/captcha).
3. Add server-side detection for suspicious automation patterns.
4. Consider redesigning scoring so brute-force streaking is not practical.
5. Keep secrets out of shipped compose files in production deployments.
