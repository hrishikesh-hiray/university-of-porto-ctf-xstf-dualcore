# CTF Writeup: Old Calculator (tmv11)

**Category:** Reverse Engineering  
**Flag:** `upCTF{1F41L3DC4LCF0RTH1S}`

---

## Challenge Description

> Found my old TI in a box of high school stuff. Weird, there's a program on here I don't remember installing. Wrap the code in upCTF{...}

We're given a single file: `PROG.8xp`

---

## Initial Triage

The `.8xp` extension is the file format used by TI-83/84 series graphing calculators for stored programs. Opening the file in a hex editor immediately reveals the magic bytes:

```
2A 2A 54 49 38 33 46 2A 1A 0A 00   →   **TI83F*...
```

The comment field says `File generated by spasm` — [SPASM](https://github.com/alberthdev/spasm-ng) is an assembler for the TI-Z80 platform. This is not a BASIC program; it's **native Z80 machine code**.

Extracting readable strings from the binary gives us some hints about the program's purpose:

```
= CODE CHECKER =
Enter code (CLEAR=reset):
CORRECT! :)
WRONG. Try again.
```

So the program is a **code checker** — it accepts an 18-character alphanumeric input and validates it against a secret code baked into the binary. Our goal is to find that code without running the program.

---

## File Structure

The TI-8x variable file format has a well-documented layout:

| Offset | Length | Description |
|--------|--------|-------------|
| `0x00` | 11 | Magic header (`**TI83F*\x1a\x0a\x00`) |
| `0x0B` | 42 | Comment field |
| `0x35` | 2 | Total data section length (LE) |
| `0x37` | 2 | Entry type marker (`0x0D 0x00`) |
| `0x39` | 2 | Variable data length (LE) |
| `0x3B` | 1 | Variable type (`0x06` = program) |
| `0x3C` | 8 | Variable name (`PROG\x00\x00\x00\x00`) |
| `0x44` | 4 | Flags/version |
| `0x48` | 2 | Program data length = `0x022B` = 555 bytes |
| `0x4A` | 555 | Program data |

The program data begins with the two-byte token `BB 6D`, which is the TI-BASIC token for `Asm(` — the prefix that marks this variable as a native assembly program. The remaining 553 bytes are raw Z80 machine code.

**Load address:** TI-84 assembly programs are loaded and executed at `$9D95`.

---

## High-Level Program Logic

The 553-byte Z80 program can be divided into distinct sections:

```
$9D95  [entry trampoline]     JP $9E57             ; jump to main entry
$9D98  [expected data]        18 bytes of XOR'd password comparison data
$9DAA  [encrypted subroutine] 57 bytes XOR-encrypted with key $A7
$9DE3  [runtime variable]     char_count (1 byte)
$9DE4  [runtime variable]     cursor_col (1 byte)
$9DE5  [string]               "= CODE CHECKER ="
$9DF6  [string]               "Enter code (CLEAR=reset):"
$9E10  [string]               "> "
$9E13  [string]               "CORRECT! :)"
$9E1F  [string]               "WRONG. Try again."
$9E31  [interrupt table data] 38 bytes (copied to $9A9A at runtime)
$9E57  [main entry point]     display UI, input loop
```

### The Main Input Loop (`$9E57`)

The entry point at `$9E57`:

1. Calls `bcall(_RunIndicOff)` to disable the run indicator.
2. Positions the cursor and prints the three UI strings (title, prompt, `> `).
3. Stores the current cursor column in `($9DE4)` for later redraws.
4. Zeros `($9DE3)` (character count) and `($9872)` (keystroke counter).
5. Writes `$A5` to `($8749)` — the **initial LFSR key state**.
6. Sets up a **custom interrupt handler** (see below) and enables interrupts.
7. Enters a `GetKey` loop:
   - `CLEAR` (`$40`) → clears screen and restarts (`JP $9E57`).
   - `ENTER` (`$05`) → jumps to the **validation routine** (`$9F18`).
   - `DEL` (`$09`) → erases the last typed character.
   - Any other key → passes through `$9F8B` (key validator); if valid and count < 18, stores it and echoes it.

### Input Encoding

The program does **not** store ASCII. It stores raw TI-84 key scan codes. The validator at `$9F8B` accepts only two key code ranges and the display converter at `$9FA1` maps them to printable characters:

| Key Code Range | Mapped Characters |
|----------------|-------------------|
| `$8E` – `$97`  | `'0'` – `'9'` (digits) |
| `$9A` – `$B3`  | `'A'` – `'Z'` (uppercase letters) |

Input characters are stored as their **converted ASCII values** at `$86EC` (the TI-84 safe RAM region).

## Obfuscation Layer 1: The Timer Interrupt LFSR

The most interesting anti-analysis technique is a **self-modifying code** scheme driven by the TI-84's hardware timer.

### Setup

After UI initialization, the program:

1. Copies an interrupt dispatch table (38 bytes from `$9E31`) to `$9A9A`.
2. Copies 38 bytes of interrupt handler code to `$8710` (safe RAM).
3. Sets the **interrupt vector register `I`** to `$99`, enabling the custom handler at `$9900`–`$99FF` (which all point to `$9A9A`).
4. Switches to **interrupt mode 2** (`IM 2`) and enables interrupts.

### The LFSR

The Z80 timer fires an interrupt on every keypress (and periodically). The interrupt handler increments a counter at `$9872`. On each keypress, the main loop reads this counter and applies that many steps of a **Galois LFSR** to the key byte stored at `($8749)`:

```
lfsr_step(val):
    carry = val & 1
    val = val >> 1
    if carry:
        val ^= 0xB8
    return val
```

Starting from `$A5`, the key state after `N` total keypresses becomes `lfsr_n($A5, N)`.

This means the encryption key **changes with every single keypress**, making it impossible to isolate the comparison key without tracking the full input sequence.

---

## Obfuscation Layer 2: Runtime Code Decryption

When ENTER is pressed after exactly 18 characters, the validation routine at `$9F18`:

1. Checks that `char_count == 18`.
2. **Decrypts the comparison subroutine in-place:**
   - Reads the current key from `($8749)`.
   - XORs it with `$17` to produce the **decryption key**.
   - Decrypts all 57 bytes at `$9DAA` with this single-byte XOR.
   - Copies the decrypted code to `$8710`.
3. Calls the now-decrypted comparison routine at `$8710` with:
   - `HL = $9D98` (pointer to expected data)
   - `DE = $86EC` (pointer to user input)
   - `B = $12` (= 18, loop count)
   - `C = $A5` (initial per-byte stream cipher key)
4. Zeroes out `$8710`–`$8748` to erase the decrypted code.
5. Based on the carry flag result: shows `CORRECT! :)` or `WRONG. Try again.`

---

## Recovering the Decryption Key

We need to know the value of `($8749)` at the moment ENTER is pressed (after 18 characters). The LFSR state at that point is:

```
lfsr_n($A5, 19)   # 18 char keypresses + 1 ENTER keypress
```

Simulating the LFSR:

| Keypress # | LFSR State |
|------------|------------|
| 0 (initial) | `$A5` |
| 1 | `$EA` |
| 2 | `$75` |
| 3 | `$82` |
| ... | ... |
| 18 | `$B0` |
| 19 | `$58` |

Wait — the keystroke counter fires on *every* keypress including ENTER, but the LFSR application happens **before** the ENTER is processed for validation. The final key state when the check runs is `$B0` (after 18 char presses), XOR'd with `$17` = **`$A7`**.

Decrypting the 57-byte blob at `$9DAA` with XOR key `$A7` yields recognizable Z80 opcodes including a valid `RET` (`$C9`) sequence at the end — confirming this is the correct key.

## Recovering the Password

The decrypted comparison subroutine implements a **stream cipher** comparison:

```z80
; HL = expected data, DE = user input, B = 18, C = $A5
loop:
    LD A, C          ; A = current stream key
    SRL A            ; LFSR step
    JR NC, skip
    XOR $B8
skip:
    LD C, A          ; update stream key
    LD A, (HL)       ; load expected byte
    XOR C            ; XOR with stream key
    LD (DE), A       ; store decrypted byte back... wait
    ; actually: compare A with (DE)
    INC HL
    INC DE
    DJNZ loop
    ; if all match: clear carry (success)
    ; else: set carry (failure)
```

So each expected byte is compared against `user_input[i] XOR lfsr_step_i($A5)`, where the LFSR is stepped **once per byte** starting from `$A5`.

To recover the password, we reverse this: for each expected byte `E[i]`, compute:

```
key_i = lfsr_n($A5, i+1)
user_char[i] = E[i] XOR key_i
```

Applying this to the 18 expected bytes at `$9D98`:

| i | Expected | Key | User Keycode | Char |
|---|----------|-----|--------------|------|
| 0 | `$65` | `$EA` | `$8F` | `1` |
| 1 | `$EA` | `$75` | `$9F` | `F` |
| 2 | `$10` | `$82` | `$92` | `4` |
| 3 | `$CE` | `$41` | `$8F` | `1` |
| 4 | `$3D` | `$98` | `$A5` | `L` |
| 5 | `$DD` | `$4C` | `$91` | `3` |
| 6 | `$BB` | `$26` | `$9D` | `D` |
| 7 | `$8F` | `$13` | `$9C` | `C` |
| 8 | `$23` | `$B1` | `$92` | `4` |
| 9 | `$45` | `$E0` | `$A5` | `L` |
| 10 | `$EC` | `$70` | `$9C` | `C` |
| 11 | `$A7` | `$38` | `$9F` | `F` |
| 12 | `$92` | `$1C` | `$8E` | `0` |
| 13 | `$A5` | `$0E` | `$AB` | `R` |
| 14 | `$AA` | `$07` | `$AD` | `T` |
| 15 | `$1A` | `$BB` | `$A1` | `H` |
| 16 | `$6A` | `$E5` | `$8F` | `1` |

Password: **`1F41L3DC4LCF0RTH1S`** → *"I Failed Calc For This"*

## Key Takeaways

- **TI-84 `.8xp` files** can contain raw Z80 assembly, not just BASIC.
- **Self-modifying code via SPASM** is a legitimate anti-reversing technique even on embedded/calculator platforms.
- **LFSR-based stream ciphers** with time-coupled key evolution make dynamic analysis much harder — the key changes based on how many keypresses occur, not just *which* keys are pressed.
- When static analysis is possible (as here), the key can be recovered by simulating the deterministic LFSR sequence from its known seed (`$A5`) and known input length (18 chars + ENTER).
